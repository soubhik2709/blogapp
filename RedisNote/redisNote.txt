6.🧠 What Is Eviction Policy?
ans:
When Redis memory limit is reached:
Max memory reached 🚨
Redis must decide:
👉 Which key should be removed?
That decision rule is called:
📌 Eviction Policy
It is simply a strategy to free memory.
maxmemory-policy allkeys-lru



7.How does Redis know which key is “least recently used”?
ans:
Redis tracks key access internally.
Every time you:
GET blog:1
Redis updates metadata:
Last access time
Internal counter
It does NOT store full timestamps for every key (too heavy).
Instead:
👉 Redis uses an approximate LRU algorithm
It keeps small sampling info to estimate least used keys.

example
User activity:

10:00 → search:react
10:01 → search:node
10:02 → search:redis
10:10 → search:react again

Now which key is oldest unused?
node
redis

If new memory needed:
👉 One of those gets removed.
Because:
react was accessed recently at 10:10.


8. Important: LRU Is Not Exact
ans:
Redis does not maintain perfect order list (too expensive).
Instead:
It randomly samples few keys
Chooses least recently used among them
This makes it:

⚡ Super fast
⚡ Scalable

Tradeoff: small approximation error (which is fine)


📊 Other Eviction Policies
There are multiple:
Policy	        Meaning
noeviction	    Throw error when memory full
allkeys-lru	    Remove least recently used from all keys
volatile-lru	Remove least recently used ONLY among keys with TTL
allkeys-random	Remove random key
volatile-ttl	Remove key with shortest TTL


TTL:
→ Deletes after time expires
LRU:
→ Deletes when memory is full

2.📦 Redis List Basics (Internally)
Redis List is:
      Ordered
      Allows duplicates
      Supports push/pop from both ends
      Very fast (O(1) operations)
      Think of it like this:
      
      [ job3, job2, job1 ]
        ↑               ↑
       left           right


3.LPUSH-Add Task to Left Side,Adds a value to the LEFT side of the list.Every new item goes to the left front. for pop use  BRPOP

4.RPUSH - reverse of LPUSH. for pop use BLPOP.

5.BRPOP  - Blocking Pop . Blocking means - If queue empty →  It WAITS,blocks,It does NOT return until a job arrives. where  in RPOP returns null immediately.This is PERFECT for background workers. No matter use of BRPOP or BLPOP just maintain the stack, queue order.

6.LLEN - list length , Check Queue/stack Size.


7.(FIFO Pattern)
Producer → LPUSH - LPUSH adds to left
Worker → BRPOP -BRPOP removes from right,So oldest job is processed first.O(1)

  Priority Jobs  - critical password reset email -> RPUSH queue normalJob
                   Instead of RPUSH--->             LPUSH queue urgentJob
